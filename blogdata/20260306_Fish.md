MSYS2 上的 Rust 和 Fish 4 纪实。

## 起因

见上一篇。

## Rust 的 Cygwin target

首先是给 Rust 加上 `x86_64-pc-cygwin` 这个 [target](https://github.com/rust-lang/rust/pull/134999)。一开始想的是加两个，一个给原版的 Cygwin，一个给修改版的 MSYS2。但是其实 MSYS2 只是改了一下 import lib 还有 DLL 的名字，ABI 是没有变化的，这样做有点浪费。Rust 的维护者纠结了一下要不要把它做成 `x86_64-pc-windows-cygwin`，但是这意味着 `target_os = "windows"`。虽然事实如此，但是对代码维护来说是一个灾难。我希望它是 `target_os = "cygwin"`，然后属于 `unix`，这样可以重用许多适用于 POSIX 的代码。

于是维护者让我开了一个 [MCP](https://github.com/rust-lang/compiler-team/issues/826)。经过了一些讨论之后，投票通过了我的提案。所以 `"cygwin"` 成了一个新的 `target_os`，而且属于 `unix`。顺便加上了 `cc-rs` 的 patch。

同时间去 LLVM 开了个 [PR](https://github.com/llvm/llvm-project/pull/121439) 修复 DSO local 的问题。后来还发现参考的那个日本博客还提到了这个 PR，不知道为什么作者不自己来呢？

然后是把大猫的工作移植过来。他的大部分代码可以重用，少部分由于 Rust std 不断重构，需要手动处理 diff。由于 Rust std 还有一部分 `rust-lang` 下面项目的其它依赖，所以还去 patch 了 `libc` 和 `compiler-builtins`。这一个 PR 持续了大概三个礼拜，不算太快也不算太慢，合并了之后能正常编译普通的 Cygwin 程序了。

## 修复 Cygwin target 的 bug

Cygwin target 的 stack overflow 处理是坏的。大猫根本没管这茬，但是我觉得还是要处理一下。用 POSIX 的信号处理机制根本不行，因为这其实是 Windows，哪来的信号。最后干脆把 Windows 的相关代码复制了过来，直接调用 `AddVectoredExceptionHandler`。

运行 gdb 的时候发现线程名字是截断的，发现 Cygwin 故意截断了，模仿 Linux。于是加上了相关 [patch](https://github.com/rust-lang/rust/pull/138765)。

这段时间就已经可以在加上了一些 patch 的情况下编译 fish 了。之前就发现了 fish 调用 `posix_spawn` 会返回 `EINVAL`。Fish 的维护者坚持认为需要 `POSIX_SPAWN_SETSIGDEF`，于是去问 Cygwin 的维护者。维护者说确实是个 bug，但是是 newlib 的。发的 [patch](https://inbox.sourceware.org/newlib/TYCPR01MB10926A7C1374B9CB3F7D493C8F8D12@TYCPR01MB10926.jpnprd01.prod.outlook.com/) 也是很快合并了。

于是也给 Cygwin target 加上了 `posix_spawn` [支持](https://github.com/rust-lang/rust/pull/139617)。Cygwin 不支持弱符号，但是支持的 POSIX 很新，所以可以直接强链接 `posix_spawn_file_actions_addchdir_np`。

另外一个坑是 [backtrace](https://github.com/rust-lang/backtrace-rs/pull/704) 支持。Cygwin 是一个基本上在用 GNU 套件的 POSIX target，但是基础仍然是 Windows，二进制格式仍然是 PE。于是选择使用 Windows 相同的 API 获取调试符号信息。同时学会了一个 Cygwin 自己的 API `cygwin_conv_path`，可以用来在 POSIX path 和 Win32 path 之间互相转换。

六月前后，Zulip 上收到了一个安全报告。由于没注意到上面说了不要马上发 PR，先讨论一下解决方案，手快就直接开了个 [PR](https://github.com/rust-lang/rust/pull/141864)。还好当时还没什么人用这个 target。这个是有关路径处理的，由于 cygwin 的 API 其实大部分同时支持两种格式的路径，而 Rust 只处理了 POSIX 的一种，导致会出现问题。解决方案搞了很久，最后借助 `cygwin_conv_path` 勉强实现了。

## Fish 的 Cygwin 正式支持

Rust 的 Cygwin target 基本完成之后，fish 就可以[支持](https://github.com/fish-shell/fish-shell/pull/11238) Cygwin 了，虽然需要在 Windows 上通过 `-Z build-std` 才能编译，但是可以写出来 `PKGBUILD` 了。

这个 PR 主要解决的是 `wchar_t` 的问题。在几乎所有 POSIX 系统里面，它是32位的，但是在 Cygwin 里面和 Windows 一样，是16位的。于是许多函数不太能用了，因为现在的 fish 在用 Rust 的 `char`。好消息是 Cygwin 支持的 C 标准很新，所以用 `c32rtomb` 和 `c32rtomb` 就行了。

## Rustc 的 Cygwin 支持

当时就想要把自己的 `PKGBUILD` 合并到 MSYS2 的仓库去，但是 MSYS2 的维护者不希望让 MSYS2 的包依赖 MINGW 的包。所以还是要想办法编译一个 Cygwin 上面跑的 rustc。

想要在 Cygwin 上编译 rustc<del>，一点也不复杂，真的</del>。只需要解决所有依赖的包的 Cygwin 支持就行了：
* [stacker](https://github.com/rust-lang/stacker/pull/122)
* [libloading](https://github.com/nagisa/rust_libloading/pull/173)
* [ctrlc](https://github.com/Detegr/rust-ctrlc/pull/131)
* [getrandom](https://github.com/rust-random/getrandom/pull/654)
  * 这个比较特殊一点是，很多包还在间接依赖 `getrandom 0.2`，所以我说服作者开了个 backport。
* [nix](https://github.com/nix-rust/nix/pull/2610)
* [rustix](https://github.com/bytecodealliance/rustix/pull/1410)
* [socket2](https://github.com/rust-lang/socket2/pull/568)
  * 这个包的维护者也是 [mio](https://github.com/tokio-rs/mio/pull/1871) 的维护者，他那段时间看起来真的很忙，一个月才冒一次泡。

当然还要感谢 MSYS2 的维护者，解决了 LLVM 和 libgit2。

那段时间正在京东实习，反正任务也不累。每天摸鱼的时候就去看看 PR 进度。多数时候啥也不会发生。

调试许多 Cygwin 二进制的时候发现 thread-local 的 bug，非常恼火。于是探寻了一下为什么不能使用更好的 TLS 机制。果然，其实可以直接[打开 emutls](https://github.com/rust-lang/rust/pull/141719)。结果 rustc 死活编译不过去，发现 bootstrap 里面居然偷偷搞奇怪优化，随[关之](https://github.com/rust-lang/rust/pull/141846)。

Bootstrap rustc 很麻烦，因为 Windows 不支持 POSIX 的路径。从 `llvm-config` 之类的程序拿到的路径没办法直接用，而 Cygwin 的 LLVM 又需要额外的补丁，没办法从 Rust 这边的源码编译。导致从 Windows 交叉编译到 Cygwin rustc 这一步基本上需要手扶，后面才能简单一些。

## `fork` 的根本难题

众所周知，Windows 上没有 `fork`。Cygwin 的做法是，试图从头启动这个进程，然后想办法填充相同的 PCB。于是，父进程加载的所有 DLL 都要在子进程加载一遍，并且要加载到相同的内存空间。然而 Windows 内核有自己的想法，那块内存空间有可能被占用了。此时 Cygwin 子进程只能崩溃，fork 失败。

又众所周知，rustc 会加载 proc macro。每一个 proc macro 实际上是一个 DLL。为了保证内存安全，proc macro 是不能卸载的。于是一个 rustc 进程加载了许多 DLL，尤其是编译 rustc 的时候。从 rustc 启动 gcc 来链接，又需要 fork + exec；即使是 posix_spawn 也会调用 fork + exec。于是 rustc 自编译的时候往往到最后会因为 fork 失败而终止。和 cygwin 维护者[讨论](https://inbox.sourceware.org/cygwin/TYCPR01MB10926F334CDC9344A89527583F89CA@TYCPR01MB10926.jpnprd01.prod.outlook.com/)了许多，但是没有什么好的解决方案。一个想法是，让 `posix_spawn` 在可能的情况下直接用 `CreateProcess`，但是能力很有限。

这个问题就留给后来者吧。

[Rust 1.89](https://github.com/msys2/MSYS2-packages/pull/5571) 正式进入了 MSYS2，然后是 [fish 4.0.2](https://github.com/msys2/MSYS2-packages/pull/5572)。其实 fish 的正式支持要 4.1，这里用了我的 patch。自此，我的工作告一段落。后续 Rust 和 fish 的更新，MSYS2 的维护者自己就能完成，而且 PR 发的比我快。

## 总结

Rust 的 Cygwin 支持是我 2025 年做过的最大的工程之一。要说另一个大工程，可能是终于发出去了一篇[论文](https://www.sciencedirect.com/science/article/pii/S0168900225007880)。这个过程让我积累了不少经验，并不是任何人都知道给 Rust 加一个新的 target 支持该怎么做。

也有一些遗憾，比如最后也没给 `compio` 加上 Cygwin 支持，要知道这可能是最容易合并的那一个……但是可见的时间内感觉并没有让 `compio` 上 Cygwin 的动力。
